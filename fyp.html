Introduction
Virtual Reality
Virtual Reality or VR, is a simulated immersive experience that mimics real environments using
computer software and hardware. Modern VR devices use a combination of a headset, sensors and
motion controllers to tap into the user’s sense of sight, hearing and touch to give a sense of
presence in a virtual space.
VR’s conception began in the 19th century with many companies trying their hand at the exotic
technology. For example, in the 1980s Atari founded a research laboratory for virtual reality. John
Larnier, considered a key figure in VR, founded VPL Research, a pioneering company developing
many devices. VPL Research has developed several VR devices like the Data Glove, which eventually
became the Power Glove. The Power Glove was an early affordable VR device but was hard to use
and not popular.
Figure 1: Power Glove, an early VR device, taken from https://en.wikipedia.org/wiki/Power_Glove#/media/File:NES-Power-
Glove.jpg
More recently the development of virtual reality has been driven by the emergence of powerful
computing and visualisation technologies and development in optical technologies. VR requires a
higher resolution and higher fps (frames per second) to give a smooth experience͘ Modern GPU’s are
capable of delivering these necessities for VR at a reasonable price, for example Nvidia’s GTX 10
series are marketed as “VR ready”͘
VR is a growing market with many major technology companies getting involved. Facebook, Google,
Sony, and Samsung are some of the major players who have dedicated VR teams. Applications of the
technology are numerous and are still being figured out. Examples of application areas are gaming,
training, medicine, and robotic. Music is also an area where VR could be interesting and is the focus
of this project. Recently Google Play and Enosis VR collaboratively produced a VR experience for
Queen’s Bohemian Rhapsody to support their tour͘ New applications, content, ease of use and
affordability are some of the factors for widespread adoption of VR in music.
This project has been undertaken to demonstrate how VR can be combined with the mainstream
music application Spotify using affordable technologies and limited resources. Widespread
development of end-user application such as this in VR using accessible technologies is expected to
drive further adoption of VR in everyday music experiences.

Spotify VR is a Unity application that offers you the experience of Spotify in Virtual Reality. It allows
you to reach out and touch your favourite songs and manipulate them in 3d space. It uses the power
of VR to give you a new perspective of your listening habits and the charts.

Why Unity?
Unity is a game engine that supports multiple platforms. It was announced in 2005 and has gone on
to be an extremely successful platform for both indie developers and triple A studios. The main
appeal of Unity is that it is free for use as long as your revenue is less than $100,000, with paid
packages offered beyond that revenue. Unity documentation is excellent due to its maturity, with
active forums where thousands of questions have been answered. Unity supports all the main VR
platforms, Oculus included. Oculus itself has documentation on how to integrate with Unity with
sample projects and code to help you get started. The other main option for a game engine that
supports 3D VR is the Unreal Engine͘ However Unreal Engine’s language is C++, which I have no
experience with. Unity uses C# which I am familiar with.

Why Use Spotify VR?
The VR market is in its infancy with high growth potential. With big names like Valve, Facebook,
Sony, Samsung and Google all getting behind VR, it looks like it’s here to stay͘ This new VR market
has a lack of content right now. Most VR experiences are short and are at the proof of concept stage.
Because of this, VR content that you can revisit like Spotify is extremely valuable. The VR market
currently has no dedicated music applications, there is no official Spotify VR support. This project
shows the potential of music in VR.

Spotify API
The Spotify API is based on REST (Representational state transfer) principles. It allows applications to
retrieve and manage Spotify data and manipulate Spotify playback. Data resources are accessed via
Page | 17
standard HTTPS requests in UTF-8 format to an endpoint. The API follows the HTTP verbs for its
actions: “GET”, “POST”, “PUT”, and “DELETE”͘
Some common parameters include the following:
Spotify URI: Resource identifier for an artist, album or track, e.g.
“spotify:track:6rqhFgbbKwnb9MLmUQDhG6”
Spotify ID: Identifier at the end of a Spotify URI, e͘g͘ “6rqhFgbbKwnb9MLmUQDhG6”
Spotify Category ID: String identifying the Spotify category, e͘g͘ “chill”
Spotify User ID: Unique string identifying a Spotify user, e͘g͘ “wizzler”
Spotify URL: An HTML link that opens a track, album, app, playlist or other Spotify resource in a
Spotify client, e.g. http://open.spotify.com/track/6rqhFgbbKwnb9MLmUQDhG6
The Spotify API features rate limiting, you cannot overload the API with too many requests.
Responses are in the form of JSON.
Used SpotifyAPI-NET to interact with the Spotify web API. SpotifyAPI-NET is written in C#/.NET which
is the same language that is used with Unity and so was a good choice for this project. It removes the
need to deal with JSON as responses are returned as objects.

Authentication
To authenticate to Spotify, you first need a client ID, which you obtain by registering your application
on “https://beta.developer.spotify.com”͘
There are three methods of authentication with SpotifyAPI-NET: ImplicitGrantAuth,
AutorizationCodeAuth and ClientCredentialsAuth. AutorizationCodeAuth requires server-side code
to run securely, and so is not recommended, especially since I will not be running my own server in
this project. ClientCredentialsAuth also requires server-side code to run securely, and so is not
recommended for the reasons previously stated. Therefore, I chose ImplicitGrantAuth as my method
for authorization.
Authentication involves creating a webApiFactory object, providing a redirect URI “http://localhost”,
a port number, a client ID, and a series of scopes you wish to access.

Figure 20: Spotify Auth Code
Scopes allow you to specify what kind of data the application will access. For this application, it
requires access to the user’s profile, library, followed artists/playlists, top tracks and recently played͘
It also needed to be able to modify the Spotify playback state e.g. play/pause music.

Spotify Charts
Loading the Spotify charts data is handled differently to the others as there is no endpoint available
with the Spotify API for requesting chart data.
Getting Spotify Charts
Spotify chart data is available at https://spotifycharts.com/regional where you can download the
charts as a csv.
Spotify Chart CSV
The following columns are in the csv:
• Position - position in the charts, e.g. 1 is the number one song
• Track Name - the name of the track
• Artist - the artist who performs the track
• Streams - the number of times the track has been streamed
• URL - the URL for playing the song e.g.
https://open.spotify.com/track/1rfofaqEpACxVEHIZBJe6W
When Unity starts, ReadCSV is called, which uses a library called CsvFileReader to read in all the rows
of the CSV. The rows are then looped through and a ChartTrack is created using the row as its input.
Each ChartTrack is then added to a list.
Page | 26
Chart Track
Every other type of Spotify data is represented by a PlaylistScript but a chart track is different, its
handled by ChartTrack and then later converted to a PlaylistScript. A ChartTrack contains 6 string
variables for storing the necessary data associated with a Spotify chart track.
The ChartTrack constructor is straight forward. It maps the items in the string list passed to it (the
CSV row) to its variables. Regex is used to extract the important part of the URL out as the URL
provided doesn’t work with Spotify PI-NET͘ If the pattern doesn’t match an error message is logged͘
Here is the pattern used by the regex.
Input to the regex: “https://open.spotify.com/track/4eWQlBRaTjPPUlzacqEeoQ”
Output: “spotify:track:4eWQlBRaTjPPUlzacqEeoQ”
The chart tracks are loaded into the scene similar to the top tracks previously mentioned. However,
there are some differences. The ChartTrack is converted to a PlaylistScript and its variables are
assigned.
Page | 27
Each chart track has a “PopularityCube” as one of its children͘ This “PopularityCube” is just a Unity
cube with a material assigned to it to make it Spotify green. The cubes scale is assigned by using
chart track streams variable. The streams variable is multiplied by a streamsScaling variable to allow
the height of the cube to be adjusted in the Unity Editor. Streams is the number of times a track has
been streamed so it provides a visual representation of a tracks popularity which translates to VR
well. Scale is a very powerful tool in VR and is better at conveying information than just numbers.
Saving and loading user Spotify data locally
At the beginning of this project, every time the Unity scene was played, all the Spotify data for the
user was loaded. This would take 10s of seconds and slowed down the workflow as well as providing
a bad experience for the user. The decision was made to save the Spotify data locally and to load it
on start-up, drastically reducing start-up time.
Serialization
Serialization is the process of saving an object or data structure into a format that can be
reconstructed later, with no changes. Unity serialization behaves differently to normal serialization,
as serializers run in a real-time game environment. The following field types can be serialized:
• Primitive data types (int, float, double, bool, string, etc.)
• Enum types
• Certain Unity built-in types: Vector2, Vector3, Vector4, Rect, Quaternion, Matrix4x4, Color,
Color32, LayerMask, AnimationCurve, Gradient, RectOffset, GUIStyle
• References to objects that derive from UnityEngine.Object
• An array of a simple field type that can be serialized
• A List<T> of a simple field type that can be serialized
The following rules must also be followed for a field in a script to be serialized:
• Is public, or has a SerializeField attribute
• Is not static
• Is not const
• Is not readonly
• Has a fieldtype that can be serialized
These rules are problematic as the PlaylistScript we want to save contains many fields that break
these rules, e.g. sprite field. The solution to this was to make a PlaylistScriptData class which has only
the necessary fields that need to be saved.

Spotify Visualization
Music visualizations have been around for a long time. Usually they consist of a line or particles
vibrating or pulsing the beat of the music. Adapting this old idea to VR proved to be challenging.
Limitations
The ideal scenario for visualizing audio in this project would be to have access to the local Spotify
audio itself. This was not feasible however as Spotify is running independently outside of Unity and
so Unity cannot access this data without a plugin to interface with Windows. There is a plugin
available on the Unity asset store but it costs ~ 200 euro. Because of this, a workaround was needed.
The solution selected was the Spotify audio analysis associated with each song.
Spotify Audio Analysis
Spotify features an audio analysis endpoint which allows you to get a detailed audio analysis is for a
single track by passing the tracks Spotify id as a parameter. The audio analysis contains a wealth of
data and the following are the most relevant:
• Duration
• Tempo
• Key
• Segments
Segments is an array containing data about that particular segment of the song with data such as:
• Start
• Duration
• Pitches (array)
• Loudness
Using this information, a music visualizer can be made for Spotify songs.
Processing Audio Analysis
The audio analysis endpoint is called when a user plays a song. The track length, tempo, beats per
second, and key are first assigned to variables.
Page | 48
The key of the song is a number, ranging from 0 to 11 corresponding to one of the 12 keys: C, C# etc.
up to B. There are two lists, keys and colours. These lists are used to map a key from a number, to a
string, and finally to a colour which will be the colour used in the visualizer.
Figure 46: Visualizer is green for the keys D-sharp and B
Figure 47: Visualizer is yellow for the key G
The array of segments contained arrays of pitches. These arrays of pitches were approximately of
size 8, which was unnecessarily large. Only one value was needed, as there were enough segments.
This was discovered through testing, when using all of the pitch values showed a visualizer that
Page | 49
changed too quickly. Smoothness is required for an aesthetically pleasing visual experience.
Therefore, the pitches per segment were averaged and stored as an array list.
Visualizing Audio Analysis
Particles used for visualization
A Unity particle system was used to visualize the Spotify audio analysis. Unity’s particle system
allows you to modify many particle properties.
Collisions are enabled as particles bouncing off surfaces proved to be an interesting effect. Lights are
also enabled as the particle system is in a dark room where real-time lighting is effective. “Use
Particle Colour” is enabled as the particle colour should match the lighting colour.
Page | 50
The particles themselves are found for free on the asset store in the “Unity Particle Pack” provided
by Unity. After some experimentation, the lightning particle was chosen.
Visualizer Room
The visualizer room is the same size as the regular room but with nothing in it but the particle
system. A door shaped opening allows the user entrance. The opening has a collider attached which
only collides with particles, this prevents the particles from spilling out into the main room.
Darkness was achieved by changing the environment lighting ambient colour to black and re-
generating the lighting with the room game-objects set to static and disabled so the lighting from
the main room didn’t interfere͘ This produces a nice effect with the real-time lighting from the
particles.
Visualization Script
Visualizing the pitch was achieved using a co-routine, which is a special Unity function. A co-routine
is a function that can suspend its execution (yield) until the given YieldInstruction finishes, with a
YieldInstruction usually being yield for a certain amount of time, e͘g͘ “yield return new
WaitForSeconds(5);” will suspend execution for 5 seconds͘ Visualizing the pitch should run until the
song ends, is changed or is paused.
Tracking the starting time was done by using the Unity Time interface. Time.time is the time in
seconds since the start of the game. TotalTime keeps track of how long the co-routine has been
running.
Page | 51
The audio analysis segments are looped through with a for loop. A start-vector and end-vector and
created using the smoothed average pitch at that segment as values. The end-vector uses the next
value (k+1) as that is what we will be moving towards.
This process is repeated for start-vector-velocity and end-vector-velocity. A different smoothing
value is used as these vectors control the particle velocity, which needs different values than the
pitch vectors. Smoothing values are used to alter the numbers to achieve appropriate visual effects,
using the raw numbers often produces poor results. These smoothing values can be adjusted in the
Unity editor on the fly, and testing allowed them to be tweaked to ideal values.
Mathf.Lerp is a Unity function which allows you to linearly interpolate between two points. Linear
interpolation is a method to construct new data points within the range of a discrete set of known
data points. In this case, it changes particle velocity and emission rate over time.
In a while loop the particles emission rate and particle velocity are altered by lerping between the
pitch and velocity’s respective start and end vectors͘ The while loop runs until
Time.deltaTime/duration = 1, i.e. it has looped for the duration of the segment. This means that it
will take the segment duration time to lerp between the values, resulting in a smooth change
between the start and end vectors. This timing is crucial as the visualizer must be in synch with the
song.

Lighting
Lighting is an expansive topic, particularly in game engines. For this project, lighting was key to
produce a realistic environment that greatly adds immersion to the VR experience. There are a few
different lighting techniques used in Unity.
Baked Lighting
Baked lighting is pre-calculated and so has no runtime overhead. Baked lighting is for static lighting,
lighting you don’t expect to change͘ Baked lighting allows for high-quality shadows and indirect
lighting. Indirect lighting calculates the entire light path, it accounts for multiple bounces of light.
This is also called global illumination, and it helps portray a more realistic environment, as objects
affect each other’s appearance͘
Figure 48: Snippet of baked lighting from Unity documentation, https://docs.unity3d.com/Manual/LightMode-Baked.html
Page | 53
Real-time Lighting
Real-time lighting is calculated and updated every frame by Unity and so is by its nature resource
intensive. The only real-time lighting used in this project is in the visualizer room, where particles
emit light as they travel through space.
Colour Space
Colour space decides how Unity calculates lighting when mixing colours or when Unity mixes colours
reading values from textures. Linear colour space is the preferred choice for realistic lighting and so
was used in this project as realism was desired. Linear colour space requires modern hardware and is
not supported on some mobile devices͘ However, this project’s platform is high end PCs as this is
already a VR requirement and therefore linear colour space can be achieved.

Ambient Occlusion
Ambient occlusion calculates how exposed each point in a scene is to ambient lighting. It adds
realism to a scene. It is particularly noticeable when objects are close to each other or in creases or
holes. Ambient occlusion is expensive performance wise, so baked ambient occlusion was used
when generating the lighting for this scene.

Usually when lighting a scene in Unity you use one of the Unity light game-objects e.g. point-light,

Multi-threading
Multi-threading is supported by Unity but is limited. You can only construct, use or compare Unity
types in the main thread. This is a safety feature, serious errors can happen when you use multiple
threads, e.g. race-conditions, deadlock. Unity does some multi-threading itself internally, the physics
engine runs on a separate thread.
In the early stages of development, calls made to Spotify blocked the main thread of the game until
they finished. This was a huge problem as hitches like that are even more jarring in VR than on a flat
screen. It can make the user nauseous, which is not a good user experience. This problem was solved
by using multi-threading for API calls. The logic for the PlaySongUri function was removed and
placed in the PlaySongURIThread function. This means that all the previous code written does not
need to be changed as they will call the same function. A thread for the PlaySongURIThread thread
function is created with the songURI as a parameter. The thread is then started.
This approach was used for all the Spotify endpoints possible e.g. play, resume, skip track etc. This
resulted in a smoother experience for the user which is one of the project’s main goals͘

after modifying an example scene that comes with the Google Daydream Unity SDK. This was a
learning experience and I brought the knowledge I gained from this into the Oculus development,
but it was time that could have been spent elsewhere. Choosing a platform and sticking to it would
have been the most efficient approach.
The saving and loading of user Spotify data was implemented about half way into the project, which
caused some issues. If this could have been decided from the get go, time would have been saved.
Better planning with Unity prefabs could have also saved some time. The shelves containing the
records were never made into a prefab. This resulted in adjusting the shelves individually, creating
extra work.
The visualizer was originally intended to work off the local audio on the PC, however this proved to
be too challenging to overcome. A lot of time was spent researching the possibilities in this area.
There were positives too. The decision to use Unity as the game engine proved to be the right
choice. Its documentation is detailed, and its Oculus VR support is excellent. The Spotify API used
allowed Spotify data to be loaded into Unity quickly with little hassle. It bypassed the JSON
responses and gave the Spotify data as C# objects which was a real time saver.

Model a 3D space

Record player interaction
The user can place a record on the vinyl player. An animation will trigger moving the vinyl into place
and the Spotify song, playlist etc will play. The needle moves in to position. The user can also remove
the vinyl, the song, playlist etc will pause and the needle moves back.
Oculus Rift integration
The Oculus SDK was used to integrate the Oculus Rift into the Unity project.
Oculus Touch controller integration
The Oculus OVRInput package in the Oculus SDK was used to integrate the Oculus Touch controllers
as input.
Teleportation Navigation
The user can teleport to any location on the floor by using the oculus touch controllers. Movement
can also be done using the analogue sticks on the touch controllers.
Spotify music visualization
Spotify music is visualized using Spotify’s audio analysis of the track in the particle visualizer room.
Figure 60: Particle visualizer room
Play and pause Spotify tracks and playlists
Spotify tracks, playlists can be played and paused either using the record player or using the left-
hand UI. Spotify artists, recommendations, and search results can also be played.

Smooth experience
A high fps is maintained. This was achieved by making optimizations using the Unity Profiler.
Responsive interactions
Interactions are quick and generate a response. An example of this is the UI confirmation message
displayed to the user when they use the follow cube.
Intuitive UI and interactions
The UI is intuitive as it is mapped to your hand and so is always available, with clear icons for buttons
that relay their functionality. Interactions are also intuitive as they respond to your physical input.
An example of this is that you can grab a vinyl and change its position, and also throw the vinyl.
Future Features
I intend to release this application to the public and if there is a desire for it, I would like to add
functionality and improve upon it. I have several features in mind:
• Adjusting charts based on country by selecting region from globe.
• Adding viral charts.
• Support for left handed use.
• Modifying playlists using physical interactions.
• More data displayed about a track in the vinyl UI.
• More options for search, e.g. limit to tracks, albums, artists.
• Artist station you can place a vinyl on, displaces artist information in 3d space.
Conclusion
The aim of this project was to bring Spotify to virtual reality in a manner that allows a unique way to
visualize and interact with music and I feel this aim has been achieved.
It was also planned to demonstrate that a novel application could be developed using readily
available and affordable products and using a consumer product - Spotify.
While there were many challenges and obstacles on the journey, the project has delivered a playable
demo that any Spotify user can enjoy. The user can enter the VR world and experience the world of
selecting Spotify tracks, then playing, listening and tracking preferences just as they would on their
own device. The technologies used were moderately priced and affordable for any committed
developer.
